## 1. Создание жесткого диска (веб-форма)

### Endpoint
```
POST /pc_components/add_hard_drive
```

### Описание
Создает новый жесткий диск в системе через веб-форму. Доступно только администраторам.

### Требования
- **Аутентификация**: Требуется (`@login_required`)
- **Права доступа**: Только администраторы (`mode == 1`)
- **Режим**: Недоступно в тестовом режиме (`TEST_MODE`)
- **Content-Type**: `application/x-www-form-urlencoded` (form-data)

### Параметры запроса (form-data)

#### Обязательные поля:
| Параметр | Тип | Описание |
|----------|-----|----------|
| `drive_type` | string | Тип диска: `HDD`, `SSD`, `NVMe` |
| `vendor_id` | integer | ID производителя (из таблицы `vendor`) |
| `model` | string | Модель диска (до 200 символов) |
| `capacity_gb` | integer | Объем диска в гигабайтах |
| `serial_number` | string | Серийный номер диска (до 100 символов) |

#### Необязательные поля:
| Параметр | Тип | Описание |
|----------|-----|----------|
| `health_check_date` | string | Дата проверки здоровья диска (формат: `YYYY-MM-DD`) |
| `power_on_count` | integer | Количество включений диска |
| `power_on_hours` | integer | Наработка в часах (время работы диска) |
| `health_status` | string | Статус здоровья: `Здоров`, `Тревога`, `Неработает` |
| `comment` | string | Комментарий (текстовое поле) |
| `interface` | string | Интерфейс подключения: `SATA`, `NVMe`, `USB`, и т.д. |
| `purchase_date` | string | Дата приобретения (формат: `YYYY-MM-DD`) |
| `purchase_cost` | string | Стоимость приобретения (десятичное число) |

### Формат ответа

#### Успешное создание:
- **HTTP Status**: 302 (Redirect)
- **Location**: `/pc_components/hard_drives`
- **Flash сообщение**: `"Жесткий диск {vendor_name} {model} успешно добавлен"`

#### Ошибки:
- **403 Forbidden**: Пользователь не является администратором
- **400 Bad Request**: Не заполнены обязательные поля
- **400 Bad Request**: Неверный формат даты или стоимости

### Пример запроса (cURL)
```bash
curl -X POST http://your-domain.com/pc_components/add_hard_drive \
  -H "Cookie: session=your_session_cookie" \
  -F "drive_type=SSD" \
  -F "vendor_id=1" \
  -F "model=Samsung 980 PRO" \
  -F "capacity_gb=1000" \
  -F "serial_number=SN123456789" \
  -F "interface=NVMe" \
  -F "health_check_date=2024-01-15" \
  -F "power_on_hours=500" \
  -F "power_on_count=100" \
  -F "health_status=Здоров" \
  -F "purchase_date=2023-06-01" \
  -F "purchase_cost=15000.00" \
  -F "comment=Основной системный диск"
```

### Пример запроса (Python requests)
```python
import requests

url = "http://your-domain.com/pc_components/add_hard_drive"
cookies = {"session": "your_session_cookie"}

data = {
    "drive_type": "SSD",
    "vendor_id": 1,
    "model": "Samsung 980 PRO",
    "capacity_gb": 1000,
    "serial_number": "SN123456789",
    "interface": "NVMe",
    "health_check_date": "2024-01-15",
    "power_on_hours": 500,
    "power_on_count": 100,
    "health_status": "Здоров",
    "purchase_date": "2023-06-01",
    "purchase_cost": "15000.00",
    "comment": "Основной системный диск"
}

response = requests.post(url, cookies=cookies, data=data)
print(response.status_code)  # 302
```

## 2. Обновление жесткого диска (веб-форма)

### Endpoint
```
POST /pc_components/edit_hard_drive/<int:drive_id>
```

### Описание
Обновляет существующий жесткий диск через веб-форму. Доступно только администраторам. При изменении параметров состояния автоматически создается запись в истории.

### Требования
- **Аутентификация**: Требуется (`@login_required`)
- **Права доступа**: Только администраторы (`mode == 1`)
- **Режим**: Недоступно в тестовом режиме (`TEST_MODE`)
- **Content-Type**: `application/x-www-form-urlencoded` (form-data)

### Параметры URL
| Параметр | Тип | Описание |
|----------|-----|----------|
| `drive_id` | integer | ID жесткого диска для редактирования |

### Параметры запроса (form-data)

#### Обязательные поля:
| Параметр | Тип | Описание |
|----------|-----|----------|
| `drive_type` | string | Тип диска: `HDD`, `SSD`, `NVMe` |
| `vendor_id` | integer | ID производителя (из таблицы `vendor`) |
| `model` | string | Модель диска (до 200 символов) |
| `capacity_gb` | integer | Объем диска в гигабайтах |
| `serial_number` | string | Серийный номер диска (до 100 символов) |

#### Необязательные поля:
| Параметр | Тип | Описание |
|----------|-----|----------|
| `health_check_date` | string | Дата проверки здоровья диска (формат: `YYYY-MM-DD`) |
| `power_on_count` | integer | Количество включений диска |
| `power_on_hours` | integer | Наработка в часах (время работы диска) |
| `health_status` | string | Статус здоровья: `Здоров`, `Тревога`, `Неработает` |
| `comment` | string | Комментарий (текстовое поле) |
| `interface` | string | Интерфейс подключения: `SATA`, `NVMe`, `USB`, и т.д. |
| `purchase_date` | string | Дата приобретения (формат: `YYYY-MM-DD`) |
| `purchase_cost` | string | Стоимость приобретения (десятичное число) |
| `active` | checkbox | Активен ли диск (передается как `"on"` если отмечен) |

### Особенности
- При изменении параметров состояния (`health_check_date`, `power_on_count`, `power_on_hours`, `health_status`) и наличии `health_check_date` автоматически создается запись в таблице `pc_hard_drive_history`
- Комментарий из формы добавляется к записи истории в формате: `"Обновление состояния. {comment}"`

### Формат ответа

#### Успешное обновление:
- **HTTP Status**: 302 (Redirect)
- **Location**: `/pc_components/hard_drives`
- **Flash сообщение**: `"Жесткий диск {vendor_name} {model} успешно обновлен"`

#### Ошибки:
- **403 Forbidden**: Пользователь не является администратором
- **404 Not Found**: Жесткий диск с указанным ID не найден
- **400 Bad Request**: Не заполнены обязательные поля
- **400 Bad Request**: Неверный формат даты или стоимости

### Пример запроса (cURL)
```bash
curl -X POST http://your-domain.com/pc_components/edit_hard_drive/123 \
  -H "Cookie: session=your_session_cookie" \
  -F "drive_type=SSD" \
  -F "vendor_id=1" \
  -F "model=Samsung 980 PRO" \
  -F "capacity_gb=1000" \
  -F "serial_number=SN123456789" \
  -F "interface=NVMe" \
  -F "health_check_date=2024-01-20" \
  -F "power_on_hours=600" \
  -F "power_on_count=120" \
  -F "health_status=Здоров" \
  -F "purchase_date=2023-06-01" \
  -F "purchase_cost=15000.00" \
  -F "comment=Обновление состояния после проверки" \
  -F "active=on"
```

### Пример запроса (Python requests)
```python
import requests

url = "http://your-domain.com/pc_components/edit_hard_drive/123"
cookies = {"session": "your_session_cookie"}

data = {
    "drive_type": "SSD",
    "vendor_id": 1,
    "model": "Samsung 980 PRO",
    "capacity_gb": 1000,
    "serial_number": "SN123456789",
    "interface": "NVMe",
    "health_check_date": "2024-01-20",
    "power_on_hours": 600,
    "power_on_count": 120,
    "health_status": "Здоров",

```

## 3. Автоматический сбор данных (JSON API)

### Endpoint
```
POST /api/hdd_collect
```

### Описание
API endpoint для автоматического приема данных о жестких дисках с Windows ПК. Поддерживает как создание новых дисков, так и обновление существующих. Диски идентифицируются по серийному номеру. **Публичный endpoint** (не требует аутентификации).

### Требования
- **Аутентификация**: Не требуется (публичный endpoint)
- **Content-Type**: `application/json`

### Параметры запроса (JSON)

#### Корневой объект:
| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| `hostname` | string | Нет | Имя компьютера, с которого собираются данные |
| `disks` | array | Да | Массив объектов с данными о дисках |

#### Объект диска в массиве `disks`:
| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| `serial_number` | string | Да | Серийный номер диска (используется для идентификации) |
| `model` | string | Да* | Модель диска (*обязательно для новых дисков) |
| `size_gb` | integer | Да* | Размер диска в ГБ (*обязательно для новых дисков) |
| `media_type` | string | Нет | Тип носителя (используется для определения типа: `SSD`, `HDD`, `NVMe`) |
| `manufacturer` | string | Нет | Производитель (если не указан, определяется по модели) |
| `interface` | string | Нет | Интерфейс подключения (`SATA`, `NVMe`, и т.д.) |
| `power_on_hours` | integer | Нет | Наработка в часах |
| `power_on_count` | integer | Нет | Количество включений |
| `health_status` | string | Нет | Статус здоровья диска. Принимает значения: `"Good"`, `"Caution"`, `"Bad"`, `"Unknown"`. Автоматически конвертируется в русские: `"Здоров"`, `"Тревога"`, `"Неработает"`, `"Неизвестно"` |

### Логика работы

#### Для существующих дисков (найдены по `serial_number`):
- Обновляются поля: `model`, `size_gb` (сохраняется как `capacity_gb`), `interface`, `power_on_hours`, `power_on_count`, `health_status`
- **Обновление производителя (`vendor`):**
  - Если в данных указан `manufacturer`, производитель обновляется
  - Если `manufacturer` не указан, но указана `model`, производитель определяется автоматически по модели
  - **Производитель автоматически создается в базе данных, если его нет** (поиск выполняется без учета регистра)
  - Поиск существующего производителя выполняется без учета регистра (case-insensitive), что предотвращает дублирование
- **Статус `health_status` автоматически конвертируется из английского в русский:**
  - `"Good"` → `"Здоров"`
  - `"Caution"` → `"Тревога"`
  - `"Bad"` → `"Неработает"`
  - `"Unknown"` → `"Неизвестно"`
- Автоматически устанавливается `health_check_date = текущая дата`
- Обновляется `comment` с информацией о hostname: `"Последний раз обнаружен на {hostname}"`
- **Автоматически создается запись в истории состояния диска (`pc_hard_drive_history`):**
  - Запись создается при каждом обновлении через API
  - Сохраняются актуальные значения: `power_on_hours`, `power_on_count`, `health_status`
  - Дата проверки устанавливается как текущая дата
  - Комментарий: `"Обновление через API"` или `"Обновление через API с {hostname}"`
  - Это позволяет отслеживать все изменения состояния диска с временными метками

#### Для новых дисков:
- Автоматически определяется `drive_type` на основе `media_type` и `model`:
  - Если в `media_type` или `model` содержится `SSD` или `SOLID` → `drive_type = "SSD"`
  - Если в `model` содержится `NVMe` или `NVME` → `drive_type = "NVMe"`
  - Иначе → `drive_type = "HDD"`
- **Автоматическое определение и создание производителя (`vendor`):**
  - **Приоритет 1:** Если указан `manufacturer` в данных → используется указанное значение
  - **Приоритет 2:** Если `manufacturer` не указан, но указана `model` → производитель определяется автоматически по модели
  - **Приоритет 3:** Если не удалось определить → используется `"Unknown"`
  - **Производитель автоматически создается в базе данных, если его нет**
  - Поиск существующего производителя выполняется **без учета регистра** (case-insensitive), что предотвращает создание дубликатов
  - Если производитель не найден, создается новый с указанным названием
- Устанавливается `health_check_date = текущая дата`
- Устанавливается `comment = "Автоматически добавлен с {hostname}"` или `"Автоматически добавлен"`
- **Статус `health_status` автоматически конвертируется из английского в русский**

### Формат ответа

#### Успешный ответ:
```json
{
  "processed": 3,
  "total": 3,
  "new": 1,
  "updated": 2
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `processed` | integer | Количество успешно обработанных дисков |
| `total` | integer | Общее количество дисков в запросе |
| `new` | integer | Количество созданных новых дисков |
| `updated` | integer | Количество обновленных существующих дисков |

**HTTP Status**: 200

### Пример запроса (cURL)
```bash
curl -X POST http://your-domain.com/api/hdd_collect \
  -H "Content-Type: application/json" \
  -d '{
    "hostname": "PC-001",
    "disks": [
      {
        "serial_number": "SN123456789",
        "model": "Samsung 980 PRO 1TB",
        "size_gb": 1000,

        "serial_number": "SN987654321",

        "interface": "SATA",
        "power_on_hours": 15000,
        "power_on_count": 500,
        "health_status": "Caution"
      }
    ]
  }'
```

### Пример запроса (Python requests)
```python

    "hostname": "PC-001",

        {
            "serial_number": "SN123456789",
```

### Автоматическое определение производителя по модели

API поддерживает автоматическое определение производителя по модели диска. Поддерживаются следующие производители:

| Признак в модели | Определяемый производитель |
|------------------|----------------------------|
| `BESHTAU` | БЕШТАУ |
| `XRAYDISK` | XrayDisk |
| `WD`, `WESTERN` | Western Digital |
| `SEAGATE`, начинается с `ST` | Seagate |
| `TOSHIBA`, начинается с `DT` | Toshiba |
| `HP`, `HEWLETT` | HP |
| `SAMSUNG` | Samsung |
| `KINGSTON` | Kingston |
| `CRUCIAL` | Crucial |
| `INTEL` | Intel |
| `SANDISK`, `SAN DISK` | SanDisk |
| `ADATA` | ADATA |
| `CORSAIR` | Corsair |
| Не определен | Unknown |

**Примечание:** Определение выполняется без учета регистра. Если производитель не может быть определен, используется `"Unknown"`.

### Обработка ошибок
- Диски без `serial_number` пропускаются
- Новые диски без `model` или `size_gb` пропускаются
- Ошибки валидации не возвращаются в ответе, но соответствующие диски не обрабатываются
- Если производитель не может быть создан (например, из-за ограничений БД), используется производитель `"Unknown"`

### Конвертация статусов здоровья

API автоматически конвертирует английские статусы в русские при сохранении:

| Английский статус (входящий) | Русский статус (в базе) |
|-------------------------------|-------------------------|
| `"Good"` | `"Здоров"` |
| `"Caution"` | `"Тревога"` |
| `"Bad"` | `"Неработает"` |
| `"Unknown"` | `"Неизвестно"` |

**Важно:** В базе данных всегда хранятся русские статусы. В интерфейсе отображаются только русские статусы, даже если в базе есть старые данные на английском (они автоматически конвертируются при отображении).

## Примечания

1. **Аутентификация**: 
   - Endpoints `/pc_components/add_hard_drive` и `/pc_components/edit_hard_drive` требуют аутентификации через сессию Flask и права администратора
   - Endpoint `/api/hdd_collect` является публичным и не требует аутентификации

2. **Права доступа**: 
   - Добавление и редактирование через веб-формы доступно только пользователям с `mode == 1` (администраторы)
   - API endpoint `/api/hdd_collect` доступен всем

3. **Тестовый режим**: 
   - В тестовом режиме (`TEST_MODE = True`) добавление и редактирование через веб-формы недоступно
   - API endpoint `/api/hdd_collect` работает в любом режиме

4. **История изменений**: 
   - При редактировании диска через веб-форму запись в историю создается только если:
     - Изменились параметры состояния (`health_check_date`, `power_on_count`, `power_on_hours`, `health_status`)
     - И указана дата проверки (`health_check_date`)
   - **При обновлении через API `/api/hdd_collect` запись в историю создается автоматически при каждом обновлении:**
     - Сохраняются актуальные значения состояния диска
     - Дата проверки устанавливается как текущая дата
     - Это позволяет отслеживать все изменения состояния диска с временными метками
     - Комментарий содержит информацию о источнике обновления (hostname, если указан)

5. **Автоматическое определение типа**: 
   - В API `/api/hdd_collect` тип диска определяется автоматически на основе `media_type` и `model`

6. **Производители (Vendors)**: 
   - **Автоматическое создание:** В API `/api/hdd_collect` производители **автоматически создаются в базе данных**, если их нет
   - **Поиск без учета регистра:** Поиск существующего производителя выполняется без учета регистра (case-insensitive), что предотвращает создание дубликатов (например, "Samsung" и "SAMSUNG" будут считаться одним производителем)
   - **Обновление для существующих дисков:** Поддерживается обновление производителя для существующих дисков:
     - Если в данных указан `manufacturer`, производитель обновляется
     - Если `manufacturer` не указан, но указана `model`, производитель определяется автоматически по модели
     - Производитель создается автоматически, если его нет в базе
   - **Автоматическое определение по модели:** Поддерживает определение следующих производителей:
     - БЕШТАУ (по признаку: BESHTAU)
     - XrayDisk (по признаку: XRAYDISK)
     - Western Digital (по признакам: WD, WESTERN)
     - Seagate (по признакам: SEAGATE, начинается с ST)
     - Toshiba (по признакам: TOSHIBA, начинается с DT)
     - HP (по признакам: HP, HEWLETT)
     - Samsung (по признаку: SAMSUNG)
     - Kingston (по признаку: KINGSTON)
     - Crucial (по признаку: CRUCIAL)
     - Intel (по признаку: INTEL)
     - SanDisk (по признакам: SANDISK, SAN DISK)
     - ADATA (по признаку: ADATA)
     - Corsair (по признаку: CORSAIR)
     - Unknown (если не удалось определить)
   - **Обработка пустых значений:** Если `manufacturer` пустой или не указан, и модель не позволяет определить производителя, используется `"Unknown"`

7. **Формат дат**: 
   - Все даты должны быть в формате `YYYY-MM-DD`

8. **Формат стоимости**: 
   - Стоимость должна быть строкой с десятичным числом (например, `"15000.00"`)

9. **Конвертация статусов**: 
   - В API `/api/hdd_collect` статусы здоровья автоматически конвертируются из английского (`"Good"`, `"Caution"`, `"Bad"`, `"Unknown"`) в русский (`"Здоров"`, `"Тревога"`, `"Неработает"`, `"Неизвестно"`)
   - В веб-формах используются только русские статусы

## Детали реализации

### Логика работы с производителями

API использует две вспомогательные функции для работы с производителями:

1. **`get_or_create_vendor(manufacturer_name)`**:
   - Ищет производителя в базе данных без учета регистра
   - Если производитель не найден, создает нового с указанным названием
   - Обрабатывает пустые значения (создает "Unknown")
   - Всегда возвращает активного производителя (`active = True`)

2. **`detect_manufacturer_by_model(model)`**:
   - Анализирует модель диска для определения производителя
   - Использует характерные признаки в названии модели
   - Возвращает стандартизированное название производителя
   - Если определить не удается, возвращает "Unknown"

### Примеры работы с производителями

#### Пример 1: Производитель указан явно
```json
{
  "serial_number": "SN123",
  "model": "Some Model",
  "size_gb": 500,
  "manufacturer": "New Manufacturer"
}
```
**Результат:** Создается новый производитель "New Manufacturer" (если его нет в базе) или используется существующий.

#### Пример 2: Производитель определяется по модели
```json
{
  "serial_number": "SN456",
  "model": "WD Blue 1TB",
  "size_gb": 1000
}
```
**Результат:** Производитель определяется как "Western Digital" по признаку "WD" в модели.

#### Пример 3: Производитель не определен
```json
{
  "serial_number": "SN789",
  "model": "Unknown Model XYZ",
  "size_gb": 250
}
```
**Результат:** Используется производитель "Unknown" (создается, если его нет в базе).

#### Пример 4: Поиск без учета регистра
Если в базе есть производитель "Samsung", а в данных приходит "SAMSUNG" или "samsung", будет использован существующий производитель "Samsung" (дубликат не создается).
